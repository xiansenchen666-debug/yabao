# 通用问题根因排查守卫

用于处理这类问题，而不仅限于 HTML：

- 页面“看起来对了”，但行为不对
- 某个功能在 A 页面正常，在 B 页面异常
- 前端交互异常、点击失效、滚动失效、遮罩拦截
- 接口返回不对、状态不同步、鉴权异常、数据残留
- 构建成功但运行异常，或开发环境与生产环境表现不一致
- 用户明确要求“不要先调样式/先别猜，直接查根因”

## 触发时机

在以下场景必须优先触发本技能：

- 用户反馈“页面滑不动 / 滚不动 / 上下不能滑动”
- 用户反馈“点击无效 / 被遮罩挡住 / 行为和另一个页面不一致”
- 用户反馈“之前能用，现在不对了”
- 用户说“不要继续微调，请直接查根因”
- 页面、接口、脚本、环境、构建链路经过多次修改后出现异常
- 同一个项目里存在一个“正常样本”和一个“异常样本”，用户要求做对比排查
- 用户要求“按根因排查，不要凭感觉修”

## 核心原则

1. 不先猜，不先做表层微调。
2. 先找“正常样本页”，再和“异常页”做差分。
3. 优先排查会影响行为、状态、数据、环境的根因，而不是先改表象。
4. 业务功能优先保留，结构替换必须避免误删真实业务逻辑。
5. 如果问题来自“旧逻辑残留”，要清掉旧依赖，而不是只改界面或文案。
6. 任何修复都要解释“为什么会坏”，不是只给结果。

## 强制排查顺序

### 第一步：确定对照样本

必须先找一个“用户确认正常”的样本作为对照，例如：

- `k12.html` 可以滚动
- `tutor.html` 不能滚动

然后比较：

- 页面结构、脚本、样式、初始化流程
- 请求链路、参数、返回值、鉴权状态
- 构建方式、运行环境、配置差异
- 是否存在旧逻辑、旧状态、旧缓存残留
- 是否存在固定层、弹层、遮罩层、异步竞态或数据覆盖

## 根因排查清单

### A. 交互层问题

重点看：

- `html` 或 `body` 是否被设置为 `overflow: hidden`
- JS 是否给 `document.body.style.overflow = 'hidden'`
- 弹窗关闭后是否没有恢复 `overflow`
- 是否有固定层、遮罩层、透明层继续吃掉点击和滚轮
- 是否有 `preventDefault()`、`stopPropagation()` 用错位置

### B. 隐藏层或旧 UI 仍在拦截行为

重点看：

- `fixed inset-0` 的遮罩层是否还在页面上
- 隐藏态是否缺少 `pointer-events: none`
- 模态框虽然视觉隐藏，但仍然覆盖页面
- 旧弹窗、旧菜单、旧侧栏是否仍在初始化

### C. 旧逻辑残留

重点看：

- 页面、组件、接口或模块已替换，但脚本仍在读旧节点
- 例如 `auth-btn`、`user-info`、`login-modal`
- `initUI()` 还在操作已删除节点
- 旧登录逻辑仍然参与初始化
- 旧定时器、旧 watcher、旧事件绑定、旧缓存仍在工作

处理原则：

- 如果这些逻辑只服务于已删除功能，优先改为“空实现”或移除调用
- 不要让旧逻辑继续影响新链路

### D. 数据/状态问题

重点看：

- 数据是否真正请求成功
- 状态是否被后续逻辑覆盖
- 本地缓存、token、cookie 是否污染了当前行为
- 正常样本和异常样本的参数、响应、权限是否一致

处理方式：

- 对比请求入参、响应、状态流转、缓存来源
- 明确“值从哪里来，被谁改掉”
- 必要时用日志或临时输出验证关键状态

### E. 页面高度/布局问题

重点看：

- 页面视觉上像长页，但内容高度刚好一屏
- 用户会感觉“滑不动”，其实是没有更多内容可滚
- 容器高度、`min-height`、`overflow`、布局壳层是否限制了滚动

### F. 环境/构建问题

重点看：

- 开发环境和线上环境变量是否一致
- 打包后路径、路由、资源引用是否变化
- 是否存在缓存、热更新残留、构建产物未更新
- 同一份代码在不同终端/浏览器/账号下是否表现不同

## 修复策略

### 能保业务逻辑，就不要重写业务逻辑

例如：

- 家教页的发布、接单、企业微信推送逻辑要保留
- 自习室的网格预约逻辑要保留
- 接口、鉴权、缓存链路如果本来可用，不要因为 UI 替换随手删掉

允许替换的通常是：

- 顶部导航 HTML
- 菜单栏相关脚本
- 页面壳层布局
- 与旧 UI 绑定的无效脚本
- 只服务于旧流程的初始化逻辑

### 替换结构时的正确做法

如果用户要求“把 A 的结构替换成和 B 完全一样”，必须：

1. 复制 B 的整段结构，不做“看起来差不多”的近似实现
2. 同步复制相关样式参数
3. 检查 A 是否还有旧脚本、旧状态、旧初始化残留
4. 只保留真正需要的差异项

## 交付要求

完成后必须向用户明确说明：

1. 根因是什么
2. 不是表层现象的地方在哪里
3. 具体改了哪些文件和哪一类逻辑
4. 如何避免下次重复走弯路
5. 如果后续要继续排查，用户应该怎样描述问题

## 推荐回复模板

```text
我先不继续猜，也不先做表层微调，这次直接按根因排查：
1. 先找一个正常样本做对照
2. 对比结构、脚本、状态、请求和环境差异
3. 先定位真正的根因，再做最小改动修复
```

## 用户沟通建议

如果用户描述不清，优先引导用户用下面格式表达：

- 现象：什么行为不对
- 对照：哪个页面、接口、账号、环境是正常的
- 预期：要和哪个样本完全一致
- 限制：不要先猜，不要先调表象，先查根因

## 禁止事项

- 不要在没做 diff 前就反复微调 `margin`、`padding`、`translate`
- 不要把“看起来差不多”当成“实现一样”
- 不要为了统一 UI 误删业务逻辑
- 不要忽略旧脚本残留
- 不要只改 HTML，不检查脚本
- 不要只看前端，不检查接口和状态
- 不要只看代码，不检查环境差异

## 示例

用户说：

> 自习室和家教兼职还是没法上下滑动，K12 可以。

正确做法：

1. 读取 `k12.html`、`tutor.html`、`study-room.html`
2. 搜 `overflow`、`pointer-events`、`fixed inset-0`、`preventDefault`、`initUI`
3. 确认是否有旧导航逻辑残留
4. 确认页面是否高度不足
5. 再修复

而不是：

- 继续猜是菜单栏字体问题
- 继续只改导航 HTML 外观

## 泛化示例

用户说：

> 接口昨天还能用，今天返回 401，但我没改这一段。

正确做法：

1. 找一个正常请求链路作为样本
2. 对比 token、header、cookie、环境变量、接口地址
3. 检查是否是旧缓存、过期鉴权、构建环境差异导致
4. 确认根因后再修复

用户说：

> 页面样式是对的，但点击按钮没有反应。

正确做法：

1. 检查按钮事件是否绑定成功
2. 检查是否被透明遮罩覆盖
3. 检查状态是否被禁用逻辑覆盖
4. 检查是否是旧组件脚本残留
