# 页面交互根因排查守卫

用于处理这类前端问题：

- 页面“看起来对了”，但行为不对
- 某个页面不能上下滚动，另一个相似页面可以
- 菜单栏、弹窗、浮层、登录逻辑替换后，页面交互异常
- 用户明确要求“不要先调样式，先查根因”

## 触发时机

在以下场景必须优先触发本技能：

- 用户反馈“页面滑不动 / 滚不动 / 上下不能滑动”
- 用户反馈“点击无效 / 被遮罩挡住 / 行为和另一个页面不一致”
- 用户说“不要继续微调 UI，请直接查根因”
- 页面经过多次导航栏、弹窗、模态框、登录逻辑替换后出现异常
- 同一个项目里存在一个“正常页”和一个“异常页”，用户要求做对比排查

## 核心原则

1. 不先猜，不先微调样式。
2. 先找“正常样本页”，再和“异常页”做差分。
3. 优先排查会影响行为的根因，而不是先改视觉参数。
4. 业务功能优先保留，导航、壳层、样式替换必须避免误删业务逻辑。
5. 如果问题来自“旧逻辑残留”，要清掉旧逻辑依赖，而不是只改 HTML 外观。

## 强制排查顺序

### 第一步：确定对照页

必须先找一个“用户确认正常”的页面作为样本页，例如：

- `k12.html` 可以滚动
- `tutor.html` 不能滚动

然后比较：

- `body` / `html` 类名和样式
- 顶部导航结构
- 主体容器 `main` / `section` 的布局方式
- 页面高度是否足够产生滚动
- 是否有固定层、弹层、遮罩层残留

### 第二步：优先排查根因关键词

必须搜索以下内容，而不是靠肉眼猜：

```text
overflow
body.style.overflow
pointer-events
fixed inset-0
preventDefault(
addEventListener('wheel'
addEventListener('touchmove'
touch-action
onscroll
auth-btn
user-info
openLoginModal
closeLoginModal
goToLogin
logout
initUI
```

## 根因排查清单

### A. 滚动被禁用

重点看：

- `html` 或 `body` 是否被设置为 `overflow: hidden`
- JS 是否给 `document.body.style.overflow = 'hidden'`
- 弹窗关闭后是否没有恢复 `overflow`

### B. 隐藏弹层仍在拦截交互

重点看：

- `fixed inset-0` 的遮罩层是否还在页面上
- 隐藏态是否缺少 `pointer-events: none`
- 模态框虽然视觉隐藏，但仍然覆盖页面

### C. 旧菜单栏逻辑残留

重点看：

- 页面导航已替换，但脚本仍在读旧节点
- 例如 `auth-btn`、`user-info`、`login-modal`
- `initUI()` 还在操作已删除节点
- 旧登录逻辑仍然参与初始化

处理原则：

- 如果这些逻辑只服务于已删除导航，优先改为“空实现”或移除调用
- 不要让旧导航脚本继续影响新导航页面

### D. 页面实际高度不足

重点看：

- 页面视觉上像长页，但内容高度刚好一屏
- 用户会感觉“滑不动”，其实是没有更多内容可滚

处理方式：

- 检查主体内容是否真的超过视口高度
- 必要时给 `main` 增加明确的最小高度
- 示例：`min-h-[calc(100vh+12rem)]`

### E. 事件被主动阻止

重点看：

- `preventDefault()` 是否错误用在滚轮、触摸或容器滚动上
- 某些全局事件监听是否拦截了默认行为

## 修复策略

### 能保业务逻辑，就不要重写业务逻辑

例如：

- 家教页的发布、接单、企业微信推送逻辑要保留
- 自习室的网格预约逻辑要保留

允许替换的通常是：

- 顶部导航 HTML
- 菜单栏相关脚本
- 页面壳层布局
- 与旧 UI 绑定的无效脚本

### 菜单栏替换时的正确做法

如果用户要求“把 A 页菜单栏替换成和 B 页完全一样”，必须：

1. 复制 B 页导航的整段 HTML，不做“看起来差不多”的近似实现
2. 同步复制相关样式参数
3. 检查 A 页是否还有旧导航脚本残留
4. 仅保留当前页高亮状态不同

## 交付要求

完成后必须向用户明确说明：

1. 根因是什么
2. 不是表层现象的地方在哪里
3. 具体改了哪些文件和哪一类逻辑
4. 如何避免下次重复走弯路

## 推荐回复模板

```text
我先不继续微调样式，这次直接按根因排查：
1. 先找一个正常页做对照
2. 检查 overflow、固定遮罩、pointer-events、preventDefault 和旧导航脚本残留
3. 找到真正拦滚动/拦点击的层后再改
```

## 用户沟通建议

如果用户描述不清，优先引导用户用下面格式表达：

- 现象：哪个页面不能滚 / 不能点
- 对照：哪个页面是正常的
- 预期：要和哪个页面完全一致
- 限制：不要先调样式，先查根因

## 禁止事项

- 不要在没做 diff 前就反复微调 `margin`、`padding`、`translate`
- 不要把“看起来差不多”当成“实现一样”
- 不要为了统一 UI 误删业务逻辑
- 不要忽略旧脚本残留
- 不要只改 HTML，不检查脚本

## 示例

用户说：

> 自习室和家教兼职还是没法上下滑动，K12 可以。

正确做法：

1. 读取 `k12.html`、`tutor.html`、`study-room.html`
2. 搜 `overflow`、`pointer-events`、`fixed inset-0`、`preventDefault`、`initUI`
3. 确认是否有旧导航逻辑残留
4. 确认页面是否高度不足
5. 再修复

而不是：

- 继续猜是菜单栏字体问题
- 继续只改导航 HTML 外观
