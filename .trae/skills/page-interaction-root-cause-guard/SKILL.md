# 🎯 通用问题根因排查守卫 (Root Cause Guard)

当遇到顽固 Bug、功能表现不一致或用户明确要求“查找底层原因”时，**必须**遵循以下核心排查法则。严禁盲目猜测和进行表层的代码微调（如随意改样式、简单加 IF 兜底等）。

## 适用场景

- 同样的功能，在不同页面、环境、账号或设备上表现不一致。
- 问题反复出现，已经修过一次两次，但没有真正解决。
- 用户明确要求不要猜，要先定位根因再修改。
- 问题看起来在 A 点，但实际可能由 B 点、C 点或历史残留逻辑触发。

## 🧠 核心排查法则 (总结为 4 步)

1. **✋ 停止表层修补 (Stop Patching)**
   - 拒绝“头痛医头，脚痛医脚”的补丁式修复。
   - 如果同一个问题尝试修复 2 次仍未解决，必须立即停止局部微调，退回全局视角重新审视。
   - 先回答两个问题：现象到底是什么，触发条件到底是什么。不要把“猜测”当成“事实”。

2. **⚖️ 引入对照组分析 (Comparative Analysis)**
   - 寻找一个**正常运作的同类样本**（如：正常的页面、成功的接口请求、其他环境下的正确表现）。
   - 严格对比异常样本与基准样本在 数据流、生命周期、DOM结构、依赖配置 等方面的实质性差异。
   - 对比时优先看真正决定结果的因素，不要只盯着表面代码是否长得像。

3. **🔍 穿透式全链路追踪 (Deep-Dive Tracing)**
   - 打破单点思维，进行上下游排查：
     - **表现层**：不仅查代码，还要查是否被不可见元素拦截（如透明遮罩、层级覆盖）。
     - **逻辑层**：查异步竞态条件、状态流转是否异常。
     - **底层/环境**：查缓存干扰、废弃的全局变量、依赖冲突或构建配置差异。
   - 需要时按“输入 -> 处理 -> 输出”的链路逐段验证，定位问题究竟卡在哪一层。

4. **🧹 清理历史残留 (Clean Leftovers)**
   - 绝大多数“灵异”问题源于旧逻辑的暗中干扰。
   - 必须重点排查：未解绑的监听器、废弃但仍在执行的旧函数、无用的拦截器等，并在修复时一并彻底移除。
   - 修复时不仅要让当前问题消失，还要避免同类问题继续被旧逻辑反复触发。

## 排查时的执行约束

- 先复现，再下结论；不能稳定复现时，要先说明触发条件和不确定点。
- 先找差异，再做修改；没有对照分析，不要直接大改代码。
- 先定根因，再提方案；方案必须对应根因，不能只缓解表象。
- 修改后要回归验证；确认主问题消失，同时检查是否引入新的副作用。

## 🚀 AI 执行标准 (SOP)

当你进入排查模式时，你的输出必须包含以下结构：
1. **【异常排查思路】**：说明你找了哪个“对照样本”，并排除了哪些表层原因。
2. **【根因揭秘】**：一针见血地指出深层原因（如：被旧脚本拦截了事件、环境变量未生效等）。
3. **【彻底修复】**：提供直击根因的修复方案，并清理相关的历史残留代码。
4. **【验证结果】**：说明修复后如何验证，是否还有风险点或待确认项。
